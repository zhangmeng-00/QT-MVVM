# 📐 Page / ViewModel / Widget 拆分决策表

> **适用范围**：MediatorMVAQt 架构
>  **目标**：统一 UI 拆分标准，避免职责混乱与架构退化
>  **核心原则**：**拆不拆，不看界面大小，只看职责 / 生命周期 / 复用性**

------

## 1️⃣ 三个核心概念定义（先对齐）

### Page（页面）

> **用户可感知的、具有独立业务意义的功能单元**

- 通常对应一个 Tab / 页面 / Stack
- 承载一个或多个 Widget
- **拥有一个主 ViewModel**

------

### ViewModel（视图模型）

> **业务状态 + 用户意图的“翻译层”**

- 订阅 `state/*`、`event/*`
- 发布 `cmd/*`
- 暴露 `Q_PROPERTY` / `ICommand`
- **不包含 UI 控件**

------

### Widget（组件）

> **纯 UI 组件，用于显示和交互**

- 只做 Binding
- 不参与系统通信
- 不保存业务状态

------

## 2️⃣ 什么时候该拆 Page？

### 判断标准

| 判断问题                                | 回答为「是」 |
| --------------------------------------- | ------------ |
| 用户能否把它当成一个独立功能理解？      | ✔ 拆 Page    |
| 能否作为一个独立 Tab / 页面存在？       | ✔ 拆 Page    |
| 是否拥有自己的一组业务状态？            | ✔ 拆 Page    |
| 是否可能被整体替换（不同设备 / 型号）？ | ✔ 拆 Page    |

### 典型示例

- 压力控制页
- 温控页
- IO 监控页
- 测试流程页
- 参数配置页

### ❌ 不该拆 Page

- 只是页面的一部分
- 只是某个功能的子区域
- 无法独立存在

------

## 3️⃣ 什么时候该拆 ViewModel？

### 判断标准（满足任意一条 → 拆）

| 判断问题                            | 是 / 否 |
| ----------------------------------- | ------- |
| 是否有独立生命周期？                | ✔ 拆    |
| 是否可能被多个 Page / Widget 复用？ | ✔ 拆    |
| 是否订阅不同模块的 Topic？          | ✔ 拆    |
| 是否有独立的一组 Command？          | ✔ 拆    |
| 是否可以被单独测试？                | ✔ 拆    |

### 典型示例

- `PressureViewModel`
- `AlarmViewModel`（跨页面）
- `TestFlowViewModel`
- `LogViewModel`

### ❌ 不该拆 ViewModel

- 只是同一数据的不同显示方式
- 没有独立 Command
- 只是 UI 结构差异

👉 **这种情况：拆 Widget，不拆 VM**

------

## 4️⃣ 什么时候该拆 Widget？

### 判断标准

| 判断问题                 | 回答为「是」 |
| ------------------------ | ------------ |
| 是否只是 UI 的一部分？   | ✔ 拆 Widget  |
| 是否为了复用界面？       | ✔ 拆 Widget  |
| 是否为了让 Page 更清爽？ | ✔ 拆 Widget  |
| 是否不包含业务状态？     | ✔ 拆 Widget  |

### 典型示例

- 图表区域
- 参数输入区
- 按钮操作区
- 状态指示灯面板

### ❌ 不该拆 Widget

- 包含业务决策
- 直接 Publish / Subscribe Topic
- 操作 Model

👉 **这种情况：应拆 ViewModel 或 Page**

------

## 5️⃣ 三者边界铁律（必须遵守）

### 🚫 Widget 禁止

- Subscribe Topic
- Publish Topic
- 知道 Mediator
- 操作 Model

------

### ⚠️ ViewModel 允许

- Subscribe `state/*` / `event/*`
- Publish `cmd/*`
- 持有 `ICommand`
- 暴露 `Q_PROPERTY`

------

### ✅ Page 只负责

- 创建 / 注入 ViewModel
- 组合 Widget
- 不处理业务逻辑

------

## 6️⃣ 快速决策流程（文字版）

```
这是一个用户可感知的独立功能吗？
 ├─ 是 → Page
 └─ 否
     ↓
是否有独立状态 / 命令 / 生命周期？
 ├─ 是 → ViewModel
 └─ 否
     ↓
是否只是 UI 展示或结构？
 └─ 是 → Widget
```

------

## 7️⃣ 终极判断问题（最重要）

> **如果这个对象“消失了”，系统状态是否应该改变？**

| 消失对象  | 系统状态是否应变化 |
| --------- | ------------------ |
| Widget    | ❌ 不应该           |
| Page      | ❌ 通常不应该       |
| ViewModel | ⚠️ 视场景           |
| Model     | ✅ 一定会           |

------

## 8️⃣ 推荐标准结构示例

```
PressurePage
 ├─ PressureViewModel
 ├─ PressureChartWidget
 ├─ PressureControlWidget
 └─ PressureStatusWidget
```

------

## 9️⃣ 一句话结论（写给维护者）

> **Widget 是视图，
>  ViewModel 是翻译，
>  Page 是功能单元。**
